# Research Question 1
## Ключевой инсайт: асимметрия — это и есть суть сравнения

Каждая парадигма **по определению** работает с разным информационным каналом:

| Парадигма | Что нужно на входе | Constraint |
|---|---|---|
| Nonparametric (RAG) | Корпус + индекс | Качество retrieval |
| Supervised parametric (QLoRA) | Размеченные QA пары | Goldset дорого создавать, масштаб ограничен |
| Unsupervised parametric (Doc-to-LoRA) | Сырые документы | Ёмкость гиперсети, качество генерируемых адаптеров |
| Hybrid | И то, и другое | Суммарная latency |

Это **центральный trade-off**, ради которого ведется сравнение. Supervised LoRA знает мало фактов, но знает их точно через human supervision. Doc-to-LoRA видит больше, но качество "впечатывания" — открытый вопрос. RAG не учится вообще, зато точно цитирует.

## Переформулированный Research Question

> **На consumer hardware с фиксированным корпусом и ограниченным human-annotated goldset: как соотносятся четыре парадигмы knowledge injection — nonparametric retrieval, supervised parametric adaptation (QLoRA), unsupervised hypernetwork-based parametric adaptation (Doc-to-LoRA), и hybrid — по качеству, grounding и latency для document-grounded QA, когда каждая парадигма оперирует в рамках своих естественных информационных и вычислительных ограничений?**

Сокращённо для заголовка:

> **Nonparametric, Supervised Parametric, Unsupervised Parametric, and Hybrid Knowledge Injection for Document-Grounded QA on Consumer Hardware**

## Пересмотренный набор систем

| ID | System | Парадигма | Информационный вход | Что учим |
|----|--------|-----------|---------------------|----------|
| S1 | Classical RAG | Nonparametric | Весь корпус через индекс | Ничего |
| S2 | QLoRA fine-tuned | Supervised parametric | 150 QA пар (RAFT-style: вопрос + gold chunks → ответ) | Формат, стиль, работа с контекстом |
| S3 | Doc-to-LoRA | Unsupervised parametric | Документы через гиперсеть | Фактическое содержание документов |
| S4 | Cluster-routed Doc-to-LoRA | Unsupervised parametric + routing | Документы по кластерам через гиперсеть | Фактическое содержание по кластерам |
| S5 | Hybrid: RAG + best adapter + HyDE | Hybrid | Оба канала | Зависит от выбранного адаптера |

В этой постановке S2 (supervised QLoRA) отвечает на конкретный вопрос: **стоит ли вкладываться в создание goldset, или unsupervised doc-to-LoRA даёт comparable quality бесплатно?** Это практически важный вопрос — если doc-to-LoRA побеждает, то ручная разметка QA для domain adaptation бессмысленна.

При этом S2 тренируется в **RAFT-style** (вопрос + релевантные чанки → ответ), а не closed-book. Это честно: модель учится работать с контекстом, а не запоминать 150 фактов наизусть. И это тот формат, который реально используют в production RAG + fine-tuning пайплайнах.

### По ёмкости doc-to-LoRA

Даже gold-only страницы (~200-300 уникальных страниц из gold_retrieval) при ~500-1000 токенов/страницу дают 100-300K токенов. Это всё равно больше ~32K ёмкости гиперсети.

Значит, **и для S3, и для S4 нужна стратегия обработки**: чанкинг + LoRA merge, или кластеризация.

Можно выстроить gradient:
- **S3**: документы обрабатываются чанками последовательно → LoRA merge (sum/average) → один адаптер. Это "наивный" подход.
- **S4**: документы кластеризованы → per-cluster чанки → per-cluster LoRA → routing. Это "умный" подход.

## Гипотезы (обновлённые)

**H1.** Hybrid (S5) даст лучший практический trade-off.

**H2.** Unsupervised Doc-to-LoRA (S3/S4) будет сопоставим или лучше supervised QLoRA (S2) на вопросах, ответы на которые **не входят** в goldset (т.е. на тех фактах, которых S2 просто не видел). На вопросах из goldset S2 может быть точнее.

**H3.** Cluster-routed (S4) обойдёт single merged (S3), потому что merge LoRA теряет информацию, а routing сохраняет специализацию.

**H4.** RAG (S1) будет доминировать на deterministic lookup-вопросах (date, number, name), где нужно точное извлечение, а не "впечатанное" знание.

**H5.** Supervised QLoRA (S2) покажет лучшую калибровку формата ответов (меньше malformed outputs), но проиграет по покрытию фактов.

## Что это даёт научно

Сравнение парадигм knowledge injection с чётким практическим выводом:

- Если doc-to-LoRA побеждает S2 → supervised goldset не нужен для domain adaptation
- Если S2 побеждает doc-to-LoRA → hypernetwork пока не заменяет supervised fine-tuning
- Если RAG побеждает оба → parametric injection пока не оправдывает себя на small corpus
- Если hybrid побеждает всех → комбинирование парадигм выгоднее любой одной

# Research Question 2

## Пункт 1: Информационная асимметрия


- Classical LoRA на 150 QA: модель видит 150 пар вопрос-ответ. Она знает только эти 150 фактов и формат ответа. Остальные ~95% информации в документах ей недоступны.
- Doc-to-LoRA на полном корпусе: гиперсеть обрабатывает весь текст документов. Модель потенциально "знает" всю содержащуюся информацию.
- RAG: индексирует все документы, на inference находит релевантные чанки.

RAG и doc-to-LoRA оперируют одним и тем же объёмом информации (полный корпус). Classical LoRA на 150 QA — нет. 

Можно было бы "выровнять" это, подав в doc-to-LoRA только страницы из gold_retrieval (т.е. те же ~150 фактов), но это искусственно калечит doc-to-LoRA.

## Пересмотренный дизайн

С учётом всего этого, вот как я вижу итоговый набор:

| ID | System | Суть | Информационный вход |
|----|--------|------|---------------------|
| S1 | Classical RAG | Retrieval по всему корпусу, без адаптации модели | Все документы (через индекс) |
| S2 | Doc-to-LoRA (single) | Гиперсеть генерирует один адаптер на весь корпус | Все документы (через гиперсеть) |
| S3 | Cluster-routed Doc-to-LoRA | Документы кластеризованы → per-cluster LoRA → routing | Все документы (по кластерам) |
| S4 | Hybrid: RAG + best D2L adapter + HyDE | Лучший адаптер (S2/S3) + retrieval + HyDE | Все документы (оба канала) |

### Новый research question

> **На consumer hardware с фиксированным корпусом: как соотносятся retrieval-based (RAG), hypernetwork-based parametric (Doc-to-LoRA), cluster-routed parametric, и hybrid стратегии knowledge injection для document-grounded QA?**

### Новые гипотезы

**H1.** Hybrid S4 даст лучший trade-off: retrieval подстраховывает адаптер, а адаптер улучшает HyDE и генерацию.

**H2.** Single Doc-to-LoRA (S2) будет ограничен ёмкостью гиперсети: 65 документов вряд ли влезут в один проход. Стратегия чанкинга + LoRA merging определит потолок качества.

**H3.** Cluster-routed Doc-to-LoRA (S3) обойдёт S2 за счёт: (a) каждый адаптер фокусируется на подмножестве корпуса, (b) снимается проблема ёмкости.

**H4.** RAG (S1) может обойти чисто параметрические S2-S3 на deterministic вопросах, где нужен точный lookup, а не "впечатанное" знание.
